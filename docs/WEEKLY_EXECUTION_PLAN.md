# ZNXP 终极选品系统每周执行计划

生成日期：2026-05-12

关联文档：

- `docs/ULTIMATE_PRODUCT_SELECTION_SYSTEM_BLUEPRINT.md`
- `docs/PRODUCT_SELECTION_PLATFORM_ROADMAP.md`

## 执行原则

这份计划按 24 周设计，目标是把当前项目从“选品 + Shopify 运营辅助”推进到“商品机会决策操作系统”的第一版旗舰形态。

执行优先级：

1. 先做决策底座：统一商品实体、信号、评分、证据、利润。
2. 再做可执行闭环：供应商、竞品、Launch、销售回流。
3. 最后做规模化：Copilot、团队协作、自动化工作流、数据质量和成本控制。

每周都必须有可验收的实际功能交付。文档、PRD、设计稿只能作为辅助产物，不能单独算完成。

每周至少满足下面一类交付：

- 后端模型、数据库迁移或回填脚本。
- 可调用 API。
- 可运行服务或任务。
- 前端页面、组件或可交互功能。
- 自动化测试、启动脚本或工程能力。

如果某项计划只产生文档，没有代码、API、页面、迁移或可运行脚本，则该项状态只能标记为“设计完成”，不能标记为“功能完成”。

## 阶段总览

| 阶段 | 周期 | 目标 | 核心交付 |
| --- | --- | --- | --- |
| Phase 0 | 第 1-2 周 | 产品和技术骨架定型 | 信息架构、数据模型、API 设计、评分模型 |
| Phase 1 | 第 3-8 周 | 决策底座上线 | Product Graph、Opportunity Score、商品情报页、利润、风险、销售回流 |
| Phase 2 | 第 9-14 周 | 竞品和供应链能力 | 竞品监控、供应商匹配、样品流程、创意拆解、AI 报告 |
| Phase 3 | 第 15-20 周 | Launch 闭环 | Pipeline、Launch Plan、素材任务、测试数据、复盘、模型回流 |
| Phase 4 | 第 21-24 周 | Agent 和团队化 | Workflow Engine、Copilot、权限、通知、周报、成本中心 |

## 第 1 周：项目启动和功能落地准备

目标：让项目可以稳定启动，并把第一批真实功能开发任务拆清楚。设计资料可以产出，但不作为功能完成。

交付：

- 项目克隆到本地。
- 前端、后端、Postgres、Redis 能启动。
- 后端依赖缺失修复。
- 种子数据脚本修复。
- Alembic migration 能跑通。
- 推荐 API 能返回数据。
- Product Graph 和 Opportunity Score 的实现任务拆分完成。

开发任务：

- 修复 `backend/requirements.txt` 中缺失依赖。
- 修复 `backend/seed_products.py` 的 Postgres boolean 写入问题。
- 启动并验证数据库、Redis、后端、前端。
- 明确下一周要新增的后端模型、migration、回填脚本、评分服务和 API 改造。

验收标准：

- `GET /health` 返回正常。
- `GET /api/v1/products/recommendations?limit=2` 返回数据。
- 种子商品可以导入。
- 下周每一项任务都对应实际代码交付。

风险：

- 不要把文档完成误判为功能完成。

## 第 2 周：基础工程治理和开发环境固化

目标：让团队能稳定开发、启动、迁移、测试。

交付：

- 后端 Dockerfile。
- 前端 Dockerfile。
- 更新 `docker-compose.yml`，支持一键启动 backend/frontend/postgres/redis。
- 明确 Python 版本要求为 3.10+ 或 3.11。
- 补齐依赖检查。
- 基础 CI 脚本：后端 import check、alembic check、前端 type/lint。

开发任务：

- 将当前手动修过的 `asyncpg`、`email-validator` 保留在依赖清单。
- 修复 `docker-compose.yml` 中前端环境变量命名和端口约定。
- 增加 `make dev` 或 `scripts/dev.sh`。

验收标准：

- 新机器执行一条命令能启动完整项目。
- 迁移能跑通。
- 种子数据能灌入。
- 前端能访问后端 API。

风险：

- 如果环境不固化，后续每周都会被启动问题拖慢。

## 第 3 周：Product Graph 数据模型落地

目标：建立统一商品实体，不再只依赖单表 `products`。

交付：

- `product_entities` 表。
- `product_sources` 表。
- `product_signals` 表。
- `score_evidence` 表。
- 现有 `products` 到 `product_entities` 的初始化脚本。

开发任务：

- 新增后端 models。
- 新增 Alembic migration。
- 编写迁移/回填脚本：为现有商品创建 entity 和 source。
- 新增基础 API：查询 entity、source、signal。
- 保留现有 `products` API，不破坏当前页面。

验收标准：

- 现有 12 条种子商品能生成对应实体。
- 一个实体能挂多个来源。
- 一个实体能挂多条信号。
- API 能返回商品实体和来源列表。

风险：

- 同款归并第一版不要过度复杂，先做规则版：标题规范化 + 平台/source_id + URL。

## 第 4 周：Opportunity Score 3.0 后端实现

目标：让推荐从静态商品分升级为可解释机会分。

交付：

- `opportunity_scores` 表。
- `app/services/scoring.py`。
- 8 维评分：需求、增长、竞争、利润、供应、内容、店铺适配、Launch 就绪。
- 风险扣分基础版。
- 评分证据结构。

开发任务：

- 把当前 `products/recommendations` 的推荐逻辑改为调用评分服务。
- 对缺失数据设置明确兜底分。
- 每次评分保存版本、输入摘要、分数、证据。
- 返回 API 中加入 `score_breakdown` 和 `evidence`。

验收标准：

- 每个推荐商品有总分和 8 维分。
- 每个维度至少有 1 条证据或缺数据说明。
- 推荐列表能按新机会分排序。
- 不影响旧商品列表搜索。

风险：

- 第一版评分宁可简单但可解释，不要用不可调的黑盒模型。

## 第 5 周：机会雷达基础版

目标：把推荐列表升级成每日机会入口。

交付：

- 新页面 `/opportunities`。
- 机会卡片组件。
- 筛选：快速上涨、低竞争高利润、适合我店铺、待验证、风险低。
- 后端 `/api/v1/opportunities`。
- Dashboard 增加机会雷达入口。

开发任务：

- 复用评分服务输出机会列表。
- 增加机会分类标签。
- 增加空状态、加载状态、错误状态。
- 机会卡片展示：机会分、趋势、利润估算、竞争强度、风险、推荐动作。

验收标准：

- 用户打开页面能看到每日优先看的商品。
- 每张卡能说明为什么推荐。
- 能一键保存到商品库或 Pipeline 占位状态。

风险：

- 页面不要做成营销大屏，要做高密度决策工具。

## 第 6 周：商品情报详情页第一版

目标：一个页面完成商品 Go/No-Go 初步判断。

交付：

- 新页面 `/intelligence/products/:id`。
- 商品摘要。
- 评分雷达。
- 证据列表。
- 来源商品列表。
- 趋势时间线基础版。
- 风险提示基础版。

开发任务：

- 后端 `/api/v1/intelligence/products/{id}`。
- 聚合 entity、sources、signals、score、evidence。
- 前端增加评分维度组件。
- 现有商品卡片跳转到新详情页。

验收标准：

- 详情页 30 秒内能看懂：能不能做、为什么、风险是什么。
- 数据不足的区块明确显示“缺少数据源”，不要静默空白。
- 从机会雷达和选品大厅都能进入详情页。

风险：

- 第一版不要一次塞满所有终局模块，先把评分和证据展示打磨好。

## 第 7 周：利润实验室第一版

目标：让每个商品能算钱。

交付：

- `profit_scenarios` 表。
- `/profit-lab` 页面。
- 商品详情内利润卡片。
- 利润 API：成本、售价、运费、平台费、支付费、广告假设、退款率。
- 输出：毛利、贡献利润、保本 ROAS、保本 CPA、建议售价。

开发任务：

- 定义 Shopify/Etsy/Amazon/TikTok Shop 平台费模板。
- 支持按商品保存多个利润场景。
- 前端支持输入成本和费用假设。
- 将利润分回写到评分服务。

验收标准：

- 用户能为商品保存一个利润模型。
- 修改成本/售价/广告假设后，利润结果实时变化。
- 机会分中的利润维度能读取利润模型。

风险：

- 不要假装利润很精确。第一版要显示假设参数和置信度。

## 第 8 周：风险雷达和店铺适配基础版

目标：补齐 Go/No-Go 决策的两个关键维度。

交付：

- `risk_findings` 表。
- 风险检测服务基础版。
- 店铺画像字段扩展。
- Store Fit Score 基础版。
- 商品详情页展示风险和店铺适配。

开发任务：

- 风险规则：品牌词、禁售词、敏感类目、物流风险、季节性、低毛利、同质化。
- 店铺画像：主营类目、客单价、禁选类目、目标毛利、发货时效要求。
- 当前店铺手动维护画像。
- 评分服务接入风险扣分和店铺适配分。

验收标准：

- 每个商品有风险等级：低/中/高。
- 每个商品对当前店铺有适配度。
- 高风险商品不会被无解释地排到前列。

风险：

- 风险规则要允许人工覆盖，避免误杀。

## 第 9 周：供应商匹配基础版

目标：把商品机会变成可采购机会。

交付：

- `supplier_offers` 表。
- 供应商报价管理。
- 商品详情页供应商区块。
- 供应商匹配 API。
- 供应商中心改造第一版。

开发任务：

- 复用现有供应商模块。
- 支持给商品实体绑定供应商报价。
- 字段：成本、MOQ、交期、评分、可定制项、来源链接、样品状态。
- 利润实验室可读取供应商成本。

验收标准：

- 商品能绑定多个供应商报价。
- 能对比报价和交期。
- 选择报价后利润模型自动更新成本。

风险：

- 图片搜同款可以后置，第一版先支持人工录入和关键词匹配。

## 第 10 周：样品和采购验证流程

目标：让供应链验证可追踪。

交付：

- `sourcing_samples` 表。
- 样品状态：待联系、已报价、已下样、运输中、已收到、质检通过、淘汰。
- 样品任务和备注。
- 供应商评分规则。

开发任务：

- 商品详情和供应商中心都能创建样品流程。
- 支持上传样品图片/视频/质检备注。
- 样品通过后更新 supply_score。

验收标准：

- 一个商品能看到所有样品进度。
- 样品结果影响供应链评分。
- 用户能筛选“供应链已验证”的商品。

风险：

- 先做状态和记录，不要一开始做复杂采购 ERP。

## 第 11 周：竞品店铺监控基础版

目标：知道谁在卖、什么时候上新、价格怎么变。

交付：

- `competitor_stores` 表。
- `competitor_products` 表。
- `competitor_events` 表。
- `/competitors` 页面。
- 基础 Shopify 店铺扫描任务。

开发任务：

- 支持添加竞品店铺 URL。
- 抓取公开商品列表、标题、价格、图片、handle、上架/更新时间。
- 记录新增、下架、改价事件。
- 商品详情页展示相关竞品。

验收标准：

- 用户能添加竞品店铺并扫描。
- 系统能识别上新和改价。
- 商品详情能显示竞品矩阵。

风险：

- 扫描频率要可控，避免触发限制。

## 第 12 周：竞品广告和创意拆解基础版

目标：把广告和素材纳入商品决策。

交付：

- `creative_assets` 表。
- Meta/Facebook 已有数据接入到素材库。
- 创意拆解 Agent 基础版。
- `/creative-intelligence` 基础页面。

开发任务：

- 将现有 Facebook 广告数据归档为 creative asset。
- 提取广告文案、标题、落地页、开始时间、平台。
- AI 输出 hook、卖点、CTA、素材形式。
- 商品详情页显示相关素材。

验收标准：

- 一个商品能看到相关广告素材。
- 每条素材能看到 AI 拆解结果。
- 创意维度能影响评分。

风险：

- 素材数据权限和来源要记录清楚。

## 第 13 周：AI 商品报告

目标：让商品情报能变成可决策报告。

交付：

- `ai_reports` 表。
- 商品报告生成 API。
- 报告中心 `/reports` 基础版。
- 商品详情页“一键生成报告”。

开发任务：

- 报告结构：摘要、机会假设、证据、竞品、利润、供应链、风险、Launch 建议。
- 保存模型、输入快照、生成时间、版本。
- 支持 Markdown 预览。

验收标准：

- 每个商品能生成一份完整报告。
- 报告内容引用系统已有结构化数据。
- 报告能用于内部选品会议。

风险：

- 报告不能编造数据，缺数据必须明确说明。

## 第 14 周：改款建议和定价策略

目标：从“发现同款”升级为“做出差异化方案”。

交付：

- 改款建议模块。
- 定价策略助手。
- 商品详情页增加差异化方案。
- 报告中加入改款和定价。

开发任务：

- 输入：评论痛点、竞品价格、供应商可定制项、目标毛利。
- 输出：颜色/材质/套装/个性化字段/包装/价格带建议。
- 定价策略接入利润实验室。

验收标准：

- 每个可做商品至少输出 3 个改款方向。
- 每个方向有预估成本影响和售价建议。
- 用户能选择一个方向进入 Launch。

风险：

- 改款建议必须基于证据，避免泛泛而谈。

## 第 15 周：Pipeline 状态流

目标：把选品从收藏变成项目管理。

交付：

- 商品 Pipeline 状态。
- 状态：发现、待验证、找供应商、打样、待上架、测试中、放量、淘汰。
- `/pipeline` 或整合到商品库。
- 负责人、截止日期、备注。

开发任务：

- 扩展 `user_products` 或新增 pipeline 表。
- 所有机会卡片支持加入 Pipeline。
- 支持状态筛选和拖动/切换。

验收标准：

- 用户能追踪每个商品推进状态。
- 商品从机会雷达进入 Pipeline 后不丢上下文。
- Pipeline 状态影响 Dashboard 统计。

风险：

- 不要做太重的项目管理，重点是商品推进。

## 第 16 周：Launch 工作台第一版

目标：把一个商品变成一次可执行测试。

交付：

- `launches` 表。
- `launch_tasks` 表。
- `/launches` 页面。
- Launch Plan 生成。

开发任务：

- 从商品详情创建 Launch。
- 字段：目标店铺、测试渠道、预算、KPI、止损线、素材清单、上架状态。
- AI 生成 7/14 天测试计划。

验收标准：

- 一个商品能创建 Launch。
- Launch 有明确预算、KPI、止损线。
- Launch 能关联商品、供应商、素材、店铺。

风险：

- 先支持手动执行和记录，不要马上自动投广告。

## 第 17 周：素材资产库和任务协作

目标：让素材生产和测试计划连接起来。

交付：

- 素材资产库完善。
- 素材任务。

- Launch 关联素材。

开发任务：

验收标准：

- Launch 能看到需要哪些素材、已有素材、缺哪些素材。
- 素材能按商品、渠道、版本筛选。
- 素材生成成本可记录。

风险：

- 素材资产要和商品实体绑定，不要散落在任务结果里。

## 第 18 周：测试数据录入和广告表现

目标：开始形成真实测试闭环。

交付：

- `launch_metrics` 表。
- 手动录入广告/销售表现。
- 指标：花费、曝光、点击、CTR、CPC、CPA、订单、GMV、ROAS、退款。
- Launch 表现面板。

开发任务：

- 支持按天录入数据。
- 支持 CSV 导入。
- 计算基础指标和趋势。
- 将测试结果同步给评分服务。

验收标准：

- Launch 能记录每日表现。
- 系统能判断是否触发止损线。
- 测试表现能影响商品状态。

风险：

- API 自动接入可以后置，先保证数据结构和手动录入闭环。

## 第 19 周：自动复盘和下一步建议

目标：让系统能判断继续、换素材、放量还是淘汰。

交付：

- Review Agent。
- Launch 复盘报告。
- 下一步动作建议。
- Pipeline 自动状态建议。

开发任务：

- 输入：Launch Plan、素材、广告表现、订单、退款、利润。
- 输出：继续测试、换素材、调价、改款、放量、淘汰。
- 保存复盘报告。

验收标准：

- 每个完成测试的 Launch 能生成复盘。
- 复盘包含数据、判断、下一步。
- 用户能接受或拒绝系统建议。

风险：

- 不要让系统自动淘汰/放量，第一版必须人工确认。

## 第 20 周：Shopify 销售回流增强

目标：让真实销售持续反哺模型。

交付：

- Shopify 订单回流增强。
- 退款/订单/商品映射到商品实体。
- 店铺画像自动更新基础版。
- 评分模型读取真实表现。

开发任务：

- 对已发布商品建立 Shopify product id 到 entity 的映射。
- 同步订单、退款、销售额、数量。
- 计算商品实际毛利和表现。
- 更新 Store Fit Engine。

验收标准：

- 已上架商品能看到真实销售表现。
- 真实表现能影响同类商品推荐。
- 店铺画像能基于历史赢家/失败品更新。

风险：

- Shopify 权限和 API 版本要处理清楚，后续应转 GraphQL Admin API。

## 第 21 周：Workflow Engine

目标：把多个 Agent 和任务串成可追踪流程。

交付：

- `workflow_runs` 表。
- `workflow_steps` 表。
- 工作流模板：发现 -> 评分 -> 竞品 -> 利润 -> 供应商 -> 风险 -> 报告 -> Launch。
- 步骤状态、重试、日志。

开发任务：

- 抽象任务执行接口。
- 每一步保存输入快照和输出 JSON。
- 支持人工确认步骤。
- 现有 `agent_tasks` 逐步接入。

验收标准：

- 用户能启动一个完整选品工作流。
- 每个步骤可查看结果和失败原因。
- 失败步骤能重试。

风险：

- 不要一次替换所有旧任务，先做一条完整流程。

## 第 22 周：AI Copilot 第一版

目标：用自然语言查询和执行核心动作。

交付：

- `/api/v1/copilot`。
- 全局 Copilot UI。
- 支持商品/机会/Launch/报告查询。
- 支持动作：加入 Pipeline、生成报告、创建 Launch。

开发任务：

- 定义可调用工具白名单。
- 做权限校验。
- Copilot 回答必须引用结构化数据。
- 关键写操作需要用户确认。

验收标准：

- 用户能问“今天最适合我店的 10 个机会”。
- 用户能问“这款为什么推荐”。
- 用户能让 Copilot 创建 Launch 草稿。

风险：

- Copilot 不能脱离数据库瞎聊，必须以系统数据为准。

## 第 23 周：团队协作、权限和通知

目标：让系统支持团队工作。

交付：

- 角色权限完善。
- 商品负责人。
- 评论/备注。
- 审批动作。
- 通知中心。

开发任务：

- 角色：Owner、Buyer、Operator、Designer、Analyst。
- 通知：竞品上新、价格变化、任务失败、Launch 达到止损线、报告生成完成。
- 商品和 Launch 支持评论。

验收标准：

- 不同角色看到不同操作权限。
- 关键动作有审批记录。
- 用户能收到重要事件通知。

风险：

- 权限要先覆盖核心写操作，不必一开始做过细。

## 第 24 周：数据质量、成本中心和旗舰版验收

目标：把系统从功能可用提升到可运营、可计费、可扩展。

交付：

- 数据质量面板。
- AI/爬虫/视频生成成本统计。
- 周报/月报。
- 旗舰版验收清单。
- 下一阶段 backlog。

开发任务：

- 记录每个数据源成功率、失败率、更新时间、覆盖量。
- 记录 AI 调用模型、token、成本、任务类型。
- 生成商品机会周报、竞品周报、Launch 复盘周报。

验收标准：

- 管理者能看到系统数据是否可靠。
- 能看到每周发现了多少机会、推进了多少 Launch、出了多少赢家。
- 能计算 AI 和爬虫成本。

风险：

- 成本中心越早做越好，否则 AI 功能越多越难控成本。
