# ZNXP SaaS 选品平台增强路线图

生成日期：2026-05-12

关联文档：

- `docs/ULTIMATE_PRODUCT_SELECTION_SYSTEM_BLUEPRINT.md`：终局产品蓝图，定义行业最强选品系统的产品架构、护城河、数据模型、AI Agent 和实施路线。
- 本文档：执行型路线图，负责把终局蓝图拆成可落地功能清单和阶段任务。

## 目标定位

把当前项目从“选品 + 上架辅助工具”升级为“跨平台商品机会情报系统”。核心目标不是简单堆功能，而是让用户能回答三个问题：

1. 什么产品现在值得做？
2. 为什么值得做，风险在哪里？
3. 从验证、找供应商、做素材、上架、投放到复盘，下一步应该自动做什么？

最终产品形态应覆盖：数据采集、机会评分、竞品监控、供应链核算、AI 决策解释、内容生产、一键上架、投放验证、经营复盘、团队协作。

## 现有能力盘点

当前项目已经具备这些基础模块：

| 模块 | 已有能力 | 评价 |
| --- | --- | --- |
| Dashboard | 总览、趋势、推荐商品 | 有入口，但决策解释和经营闭环不足 |
| 选品大厅 | 商品搜索、筛选、平台/品类/价格/评分排序 | 基础可用，缺趋势曲线、竞品密度、利润和供应链维度 |
| 商品库 | 保存商品、管理待处理商品 | 有收藏能力，缺工作流状态和团队协同 |
| 店铺管理 | Shopify 店铺接入、店铺诊断、自动推品 | 方向正确，缺店铺画像长期学习 |
| AI 运营工作台 | 店铺诊脉、自动推品、文案、图片处理、视频生成、发布 | 有雏形，需变成可编排的 Agent 流程 |
| Shopify AI | 商品同步、SEO 优化、批量状态/价格操作、视频生成 | 对 Shopify 运营有价值，需补销售回流和利润复盘 |
| 数据源 | Facebook、GMC、小红书、Shopify、供应商管理 | 数据源分散，缺统一商品实体和可信度评估 |
| 后端任务 | agent_tasks、调度器、爬虫、发布记录 | 有任务系统基础，需补队列、重试、审计、成本统计 |

## 外部趋势依据

以下公开资料对功能方向有直接参考价值：

- TikTok Shop 的 Shop Creative Hub 已经把视频、达人、GMV、订单、互动表现放在同一分析面板，说明“创意表现 -> 商品决策”会成为选品平台的关键能力。来源：https://ads.tiktok.com/help/article/about-shop-creative-hub?lang=en
- Amazon SP-API 的 Reports/Data Kiosk 支持销售、流量、搜索词、购物篮、复购、费用、广告花费、净收益等数据，说明强选品系统必须做利润和真实销售回流，而不是只看热度。来源：https://docs.aws.amazon.com/prescriptive-guidance/latest/strategy-gen-ai-selling-partner-api/data-sp-api.html
- Shopify Admin API 对订单、商品、库存、客户等数据有完整接口，但 REST Admin API 对新公共应用已转向 GraphQL Admin API，后续 Shopify 集成应以 GraphQL 为主。来源：https://shopify.dev/docs/api/admin-rest/latest/resources/order
- Google Merchant Center API 可用于商品数据管理、诊断和洞察，适合补充搜索需求、商品 Feed 健康度和 Shopping 侧竞争信号。来源：https://support.google.com/merchants/answer/14173602
- 社媒电商趋势正在走向 AI 发现、社交购物、视频/直播、全渠道体验，选品平台需要覆盖内容和渠道适配，而不是只输出商品列表。来源：https://sproutsocial.com/insights/ecommerce-trends/
- 市面上的 Product Intelligence 工具已经强调 Store Analyzer、Product Database、Supplier Finder、Competitor Tracking、AI Reports 等能力，ZNXP 要形成差异化，必须把这些能力连成闭环。参考：https://eliotron.com/

## 必加功能清单

### P0：先补齐“最强选品”的决策底座

| 功能/模块 | 要解决的问题 | 建议实现 |
| --- | --- | --- |
| 统一商品实体 Product Graph | 同一商品在 Etsy、Amazon、Shopify、TikTok、小红书、Meta 广告里重复出现，当前难以合并判断 | 建 `product_entities`、`product_signals`、`product_sources`，用标题、图片指纹、URL、价格、类目、语义向量做归一 |
| 机会评分 2.0 | 当前评分偏静态，无法解释“为什么现在做” | 拆成需求分、增长分、竞争分、利润分、供应链分、内容分、风险分，并保留每个分数来源 |
| 利润计算器 | 没有净利润就不是选品，只是看热闹 | 支持 COGS、运费、平台费、支付费、广告 CPA/ROAS、退货率、汇率、税费、仓储费，输出毛利、贡献利润、保本 ROAS |
| 供应商匹配 | 找到商品后缺少可执行供应链 | 支持图片搜同款、关键词搜供应商、1688/AliExpress/Alibaba/本地供应商录入、MOQ、交期、评价、样品状态 |
| 竞品店铺监控 | 用户需要知道谁在卖、卖得怎样、什么时候上新/改价 | 增加 Store Analyzer：抓 Shopify 店铺商品、价格、上新、折扣、热销排序、主题、页面结构、广告素材 |
| 趋势时间线 | 只看当前热度容易追高 | 对每个商品记录每日/每周信号，展示 TikTok 观看、Meta 广告数、搜索量、价格、评论、收藏、销量趋势 |
| 风险雷达 | 强选品必须避坑 | 检测侵权词、品牌词、图片版权、敏感类目、平台禁售、物流风险、季节性、低复购、差评集中点 |
| 真实销售回流 | 选品模型必须从结果学习 | Shopify 订单、退款、转化率、利润、广告花费回流到商品实体，自动更新评分权重 |

### P1：做出差异化的 AI Agent 工作流

| 功能/模块 | 要解决的问题 | 建议实现 |
| --- | --- | --- |
| 选品 Agent 工作流编排 | 当前 Agent 是单点任务，缺连续执行能力 | 建 workflow：发现机会 -> 查竞品 -> 算利润 -> 找供应商 -> 生成素材 -> 上架草稿 -> 验证计划 |
| AI 选品报告 | 用户需要能拿去决策/汇报的结论 | 每个商品生成一页报告：机会摘要、数据证据、竞品、利润、供应链、素材建议、风险、下一步 |
| 店铺画像长期记忆 | 自动推品应更懂用户店铺 | 为每个 shop 保存定位、客单价、受众、历史爆品、失败品、风格、禁选类目、利润要求 |
| 竞品创意拆解 | 只知道有广告不够，还要知道广告为什么能跑 | 抽取 hook、卖点、CTA、素材形式、落地页结构、评论痛点，生成可复用素材 brief |
| 商品改款建议 | 跨境选品不能只搬运，要差异化 | 根据差评、竞品、供应商能力生成改款：材质、颜色、组合套装、个性化字段、包装、价格带 |
| 定价策略助手 | 上架价格不能手填 | 基于竞品价格、成本、折扣、目标毛利、平台费自动生成定价和促销方案 |
| Launch Plan | 从选品到测试的执行清单 | 输出 7/14/30 天测试计划：素材数量、预算、渠道、KPI、止损线、补货判断 |
| AI Copilot 问答 | 用户需要自然语言分析数据 | 支持“最近 7 天有什么可做的刺绣宠物品？”、“这款为什么分高？”、“我的店适合吗？” |

## 评分模型建议

建议把总分改成可解释评分：

```text
Opportunity Score =
  demand_score * 0.22
+ growth_score * 0.16
+ competition_score * 0.14
+ profit_score * 0.18
+ supply_score * 0.12
+ creative_score * 0.10
+ store_fit_score * 0.08
- risk_penalty
```

各维度建议：

| 维度 | 数据来源 |
| --- | --- |
| demand_score | Google/GMC 搜索、Amazon 搜索词、TikTok 热度、小红书互动、Shopify 竞品数量 |
| growth_score | 7/14/30 天趋势、广告新增速度、视频新增速度、收藏/评论增速 |
| competition_score | 同款数量、广告主数量、价格拥挤度、头部集中度 |
| profit_score | 成本、运费、平台费、广告假设、退款率、目标售价 |
| supply_score | 供应商数量、MOQ、交期、评分、历史履约 |
| creative_score | 视频可拍性、UGC 适配、痛点明确度、素材差异化空间 |
| store_fit_score | 店铺历史品类、客单价、受众、品牌调性、库存策略 |
| risk_penalty | 侵权、禁售、物流、季节性、政策、差评密度 |

每个分数必须保存 `evidence`，不要只给一个黑盒数字。

## 迭代路线图

### 第 1 阶段：4-6 周，打牢决策底座

目标：让商品卡片从“展示商品”升级为“可决策机会”。

交付：

- 统一商品实体与信号表
- 机会评分 2.0
- 利润计算器
- 风险雷达
- 商品详情页改造
- Shopify 订单/退款/销售回流
- 供应商基础匹配

验收标准：

- 每个推荐商品都有分数拆解和证据
- 每个商品能算出目标售价、毛利、贡献利润、保本 ROAS
- 商品详情能看到趋势、竞品、供应商、风险
- 已上架商品能回流销售表现并影响后续推荐

### 第 2 阶段：6-8 周，建立竞品和内容优势

目标：让系统知道市场上谁在卖、谁在投、什么素材有效。

交付：

- 竞品店铺监控
- 广告素材库
- 创意拆解 Agent
- 评论痛点挖掘
- 商品改款建议
- AI 商品情报报告

验收标准：

- 用户能订阅 20-100 个竞品店铺
- 系统能提示竞品上新、改价、广告暴增
- 商品报告可直接用于选品会议
- 每个商品能输出 3-5 个差异化改款方向和 5-10 条素材 hook

### 第 3 阶段：6-10 周，形成 Launch 闭环

目标：从“发现商品”走到“测试商品并学习”。

交付：

- Launch 工作台
- 广告测试数据录入/API
- 素材版本和表现归因
- 自动复盘报告
- Agent 工作流编排

验收标准：

- 每个商品有明确测试计划、预算、KPI、止损线
- 测试结果能回写商品评分
- 系统能识别“放量、继续测试、暂停、淘汰”
- 用户能一键生成下一轮素材和上架优化建议

### 第 4 阶段：平台化

目标：支持团队、权限、通知、API、成本控制。

交付：

- 多角色权限
- Pipeline 和审批
- 通知中心
- API/Webhook
- 数据质量和成本中心
- 组织级报表

验收标准：

- 团队能按角色协作
- 每条数据来源可追溯
- AI/爬虫/视频成本可统计
- 能输出周报/月报/竞品报告

## 最小可行升级包

如果要先做一版“明显强很多”的版本，建议只做下面 8 个：

1. 商品机会评分 2.0
2. 利润计算器
3. 商品详情情报页
4. 供应商匹配
5. 竞品店铺监控
6. AI 商品报告
7. Shopify 销售回流
8. Launch 测试工作台

这 8 个做好后，产品会从“看商品列表”升级为“选品决策系统”。

## 技术注意事项

- 当前本机 Python 3.9 不能跑后端代码，项目应明确要求 Python 3.10+ 或全部容器化。
- `docker-compose.yml` 目前没有后端/前端 Dockerfile，建议补齐开发和生产 Dockerfile。
- 当前 `requirements.txt` 已补 `asyncpg`、`email-validator`，后续应加 CI 检查启动依赖完整性。
- 当前调度和 Agent 任务建议逐步迁到队列系统，例如 Celery/RQ/Arq + Redis，避免长任务阻塞 API。
- 所有外部数据源都要保存 `source_url`、`fetched_at`、`confidence`、`raw_payload`，方便回溯。
- 商品图片相似度建议引入 perceptual hash 或 embedding；标题相似度建议用 embedding + 规则混合。
- AI 生成内容必须带版本、模型、prompt、输入数据快照和人工确认状态。

## 决策原则

不要只做“更多数据”。最强选品平台的护城河应是：

1. 多源数据归一成一个商品机会。
2. 每个推荐都能解释证据和风险。
3. 每个商品都能算钱。
4. 每个商品都能找到供应链和改款方向。
5. 上架和投放结果会反哺模型。
