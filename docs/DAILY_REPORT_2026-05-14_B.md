## 一、今日目标

推进选品系统的解释层和使用层，让规则可读、页面可用、团队口径一致。

## 二、今日完成

### 1. 完成选品状态标准文档

已整理并明确当前 4 个选品状态的业务定义：

- `candidate / shortlisted / featured / rejected` 的定义
- 当前系统如何判定这些状态
- 每个状态的人工处理动作

实际效果：

- 团队讨论“候选、重点、主推、淘汰”时已有统一口径

这样做的好处：

- 避免不同成员对状态含义理解不一致
- 避免后续页面展示、接口输出、人工处理口径冲突
- 让选品流程从“主观判断”变成“有标准的阶段流转”

当前统一口径如下：

- `candidate`：已入池，但未进入本周重点推进范围
- `shortlisted`：已进入本周重点池
- `featured`：已进入本周头部主推池
- `rejected`：本周不做

当前人工处理建议如下：

- `candidate`
  - 继续保留观察
  - 补标签、补判断
  - 视情况升级或淘汰
- `shortlisted`
  - 重点复核
  - 必须补充明确选品理由
  - 继续和同专题商品比较
- `featured`
  - 作为本周优先主推对象
  - 必须人工确认不是泛热品误入
- `rejected`
  - 当前周期不继续推进
  - 默认不回拉，除非明确误判

### 2. 完成评分与候选池策略说明文档

已整理并明确当前评分与候选池策略：

- `final_selection_score` 的加权公式
- 权重来源与业务解释
- 自动标签的来源说明
- 每池 15 个候选的业务依据
- 为什么当前不应把系统描述成黑箱

实际效果：

- 评分逻辑、标签来源、专题池配额来源都已经可以对外讲清楚

这样做的好处：

- 解决“评分像黑箱”的问题
- 后续调整权重、标签策略、候选池容量时有统一依据
- 方便团队复盘为什么某个商品会进入候选、重点或主推

当前加权公式如下：

```text
final_selection_score =
  embroidery_fit_score * 0.30 +
  trend_score * 0.22 +
  gift_score * 0.20 +
  campaign_score * 0.13 +
  profit_score * 0.15
```

当前规则口径如下：

- 这是加权评分，不是简单加法
- 当前权重是第一版业务权重，不是数据回归权重
- 当前自动标签来源是规则初始化，不是成熟 NLP 抽取
- 当前没有正式准确率评估值
- 每池 15 个候选来自本周业务配额设计，不是技术临时写死
- 且现在已经支持配置化，不需要固定死在代码里

### 3. 完成选品策略读取接口

新增接口：

- `GET /api/v1/products/selection/policy`

接口当前可直接返回：

- 权重
- 阈值
- 标签来源
- 各专题池目标配额

实际效果：

- 策略不再只存在于文档和代码里
- 系统本身已经能读取并输出当前策略

这样做的好处：

- 文档、代码、系统输出三者口径统一
- 后续可直接接策略面板，不需要再重新整理接口
- 运维、产品、协作者可以直接读取当前真实策略，而不是靠口头同步
