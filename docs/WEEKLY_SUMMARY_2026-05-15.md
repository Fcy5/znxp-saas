# 选品系统本周精细汇报

## 一、本周工作的核心目标

本周工作的目标，不是继续堆页面，而是把当前项目从“商品列表 + 收藏能力”推进成“能按周形成明确选品结果、能解释判断逻辑、能支持人工复核、能把复核反馈反向作用到下一轮策略”的第一版选品决策系统。

这一周的所有功能，实际上都围绕同一个核心问题展开：

**系统能不能替代一部分人工选品判断，并把剩下的人工判断收进可复盘的机制里。**

---

## 二、本周最终形成的系统结果

本周已经形成以下可执行结果：

- 本周候选款：`60`
- 重点款 `shortlisted`：`12`
- 主推款 `featured`：`8`
- 候选状态 `candidate`：`40`
- 专题池数量：`4`

当前专题池为：

- `Memorial Day`
- `Father's Day`
- `Graduation`
- `Summer`

这意味着系统已经不只是能“展示商品”，而是能产出一套本周明确可执行的选品结果。

这里需要特别说明一点：

**本周候选款 `60` 不是系统总库只有 60 个商品，也不是系统能力上限只有 60。**

当前的 `60` 来自本周运营配额设定，逻辑是：

- 本周固定 4 个专题池
- 当前每个专题池的目标候选数设为 `15`
- 所以本周候选池总量为 `4 × 15 = 60`

也就是说：

- 商品总库远大于 `60`
- 当前系统只是从更大的商品池里，按本周规则筛出 `60` 个本周候选
- 后续这个数可以调整为 `80`、`100`，也可以改成按专题动态分配

当前的 `60` 本质上是：

**本周工作池容量设定，不是系统商品总量上限。**

## 三、本周完成的主要功能

### 1. 本周自动选品链路

**完成了什么**

- 建立了本周自动选品流程
- 系统可按 4 个专题池自动拉取候选商品
- 自动生成本周候选池
- 自动把候选商品压缩为重点池和主推池
- 支持重新生成本周结果

**这个功能是干嘛的**

这个功能的作用，是把原本散落在系统里的商品，收敛成“本周要重点看的结果池”。

对业务来说，真正有价值的不是系统里有多少商品，而是：

- 本周该看什么
- 本周重点推进什么
- 本周优先测试什么

所以自动选品链路的本质，不是做一个按钮，而是把原始商品数据转成经营动作。

**背后的功能逻辑**

这条链路的逻辑是：

1. 先按专题池定义本周选品方向  
2. 再从原始商品池中筛出符合专题池方向的候选商品  
3. 对候选商品补标签、打分、生成理由  
4. 再根据分数和排序结果，把商品压缩到不同状态层级  
5. 最终形成“候选 - 重点 - 主推”的本周工作结果

也就是说，这个功能的底层逻辑不是“随机挑 60 个商品”，而是：

**专题先定方向，评分负责判断，压缩逻辑负责形成经营优先级。**

**怎么使用**

进入 `/selection` 页面后，点击“生成本周候选”，系统会自动生成：

- 本周候选池
- 重点池
- 主推池
- 对应的专题池分布和状态结果

**为什么完成这个功能**

因为如果系统不能形成每周明确的结果池，它就只是商品仓库，不是选品系统。  
本周先把这条链路做出来，是为了让系统第一次具备“按周执行”的能力。

**当前边界**

当前已经做到第一版可运行，但还没有做到极致精度。  
例如 `Summer` 和 `Memorial Day` 仍可能混入少量泛热品，后续需要继续优化召回质量。

**后续怎么优化，主要从以下几个方向推进**

1. 优化专题识别规则  
   当前部分商品只因为命中了 `summer`、`memorial` 等相关词就被召回，后续需要从“词相关”升级为“商业机会相关”，也就是不仅看节日词，还要看这个商品是否真的适合该专题下的礼品需求和定制表达。

2. 强化定制刺绣信号  
   这个系统不是泛热品推荐系统，而是定制刺绣选品系统。后续需要提高以下信号的权重：
   - `name`
   - `monogram`
   - `personalized`
   - `keepsake`
   - `memorial`
   - `gift for dad`
   - `class of`
   
   同时降低“只是季节相关、但不适合做定制”的泛商品权重。

3. 增加负向过滤  
   对以下商品类型增加排除规则：
   - 纯功能型商品
   - 定制空间弱的商品
   - 没有礼品属性的商品
   - 没有人群关系表达的商品
   - 低客单且难以做溢价的商品
   
   这样可以尽早把“泛热品”挡在专题池之外，而不是等进入候选池后再靠总分清洗。

4. 增加专题召回分  
   当前已经有全局评分，但后续还需要进一步强化“专题入口判断”，也就是在进入候选池之前，先根据专题相关度、定制信号、礼品属性、人群清晰度、内容表达潜力进行一次专题排序，再决定哪些商品能进入专题池。

5. 用人工复核结果反推规则  
   当前已经有反馈闭环，下一步需要把 `Summer` 和 `Memorial Day` 误判原因持续归类，例如：
   - 定制信号弱
   - 礼品属性弱
   - 人群不清晰
   - 场景不成立
   - 只是泛季节热词
   
   然后把这些误判原因继续反推到专题召回规则和过滤规则上，逐步提升专题池纯度。

### 2. 选品评分与状态分层

**完成了什么**

- 建立了第一版 8 维评分体系
- 自动生成 `final_selection_score`
- 自动分配：
  - `candidate`
  - `shortlisted`
  - `featured`
  - `rejected`

**这个功能是干嘛的**

这个功能的作用，是让系统对商品进行判断，而不是只做数据展示。

对于选品系统来说，最关键的问题不是“商品有没有被抓进来”，而是：

- 这个商品值不值得推进
- 是先观察，还是重点看，还是直接主推
- 如果不值得做，为什么不做

所以评分和状态分层，就是系统的核心判断层。

**背后的功能逻辑**

当前评分逻辑不是单一热度排序，而是把商品机会拆成多个维度判断。  
当前第一版已经实现 8 个核心维度：

- 广告验证
- 社媒热度
- 利润空间
- 市场竞争
- 产品质量
- 趋势时效
- 受众匹配
- 刺绣适配

系统先分别计算各维度，再按权重合成最终分数，然后再基于分数和池内排序进行状态分层。

当前实际权重配置如下：

| 维度 | 当前权重 |
| --- | --- |
| 广告验证 `ad_validation` | `0.20` |
| 社媒热度 `social_heat` | `0.18` |
| 利润空间 `profit` | `0.20` |
| 市场竞争 `market_competition` | `0.12` |
| 产品质量 `product_quality` | `0.10` |
| 趋势时效 `trend_timing` | `0.05` |
| 受众匹配 `audience_fit` | `0.05` |
| 刺绣适配 `embroidery_fit` | `0.10` |

当前权重配置位置在后端：

- `backend/app/core/config.py`
- `backend/app/core/selection_config.py`

这个设计背后的逻辑是：

- 热度高，不代表能卖
- 能卖，不代表适合定制刺绣
- 适合定制，不代表适合当前专题
- 适合当前专题，不代表值得直接主推

所以必须把一个商品拆成多个判断维度，而不是只看单点指标。

**怎么使用**

- 在 `/selection` 查看整体分层结果
- 在 `/selection/library` 按状态和分数筛选
- 在商品详情页查看单品评分卡和当前状态

**为什么完成这个功能**

因为选品系统的根本价值在于“替代部分人工判断”。  
没有评分和状态分层，系统只能展示信息，不能给出判断。

**当前边界**

当前权重已经可配置，但还属于第一版业务权重，不是基于真实销售结果回归校准后的最终权重。

### 3. 自动选品理由与解释能力

**完成了什么**

- 自动生成 `selection_reason`
- 自动生成分数摘要
- 自动给出推荐状态
- 商品详情页支持查看 8 维评分卡

**这个功能是干嘛的**

这个功能的作用，是把系统的判断过程解释出来。

选品系统如果只给结果，不给原因，会出现两个问题：

1. 运营和选品人员不信这个结果  
2. 即使结果错了，也不知道错在哪里

所以自动选品理由不是一个“文案增强功能”，而是系统可信度的一部分。

**背后的功能逻辑**

系统在给商品打分后，不只是保留总分，还会保留：

- 哪些维度拉高了分数
- 哪些维度拖低了分数
- 当前推荐状态是什么
- 为什么建议这个状态

这样做的本质逻辑是：

**把分数从黑箱结果，变成可被业务团队理解的判断过程。**

**怎么使用**

- 在商品详情页查看系统生成的选品理由
- 查看 8 维分数卡
- 保存选品决策后，系统会同步刷新分数和推荐理由

**为什么完成这个功能**

因为系统只有分数没有解释，团队无法形成统一判断标准，也无法把系统结果用于协作和复盘。

**当前边界**

当前解释已经能覆盖评分逻辑，但还没有做到完全自然语言化的“多轮问答式解释”，后续可以继续增强。

### 4. 专题池与配额策略

**完成了什么**

- 固定了本周 4 个专题池
- 将专题池配额、评分权重、状态阈值从业务逻辑中抽出为配置
- 完成了动态配额第一版

**这个功能是干嘛的**

这个功能的作用，是保证本周选品不是无主题地抓商品，而是围绕明确的经营主题来做。

如果没有专题池，系统会变成“大盘热品推荐”；  
而你要做的不是泛推荐工具，而是能服务本周业务动作的选品系统。

**背后的功能逻辑**

专题池决定本周重点方向，例如节日、礼品、季节、场景。  
配额策略决定每个专题池在本周要占多少位置。  
阈值和权重决定专题池内部如何筛商品。

这一层的底层逻辑是：

- 先确定“本周看哪些机会”
- 再确定“每类机会给多少资源”
- 最后再决定“同类机会里谁优先”

所以专题池和配额不是显示层功能，而是本周经营资源分配层。

**怎么使用**

- 在 `/selection` 页面查看当前专题池和策略面板
- 系统在运行自动选品时，会按当前专题池配置和配额执行

**为什么完成这个功能**

因为如果权重、阈值、配额都写死在代码里，系统只能运行，不能被运营和策略调整。  
这周必须先把这层抽出来，后面才能继续机制化。

**当前边界**

动态配额已经有第一版，但还不是完全自适应策略。  
后续还可以继续引入专题优先级、样本质量、真实出单反馈等更强信号。

- 新增策略读取接口
- 前端可展示当前：
  - 权重
  - 状态阈值

  - 当前生效配额
  - 当前收紧规则

**这个功能是干嘛的**

这个功能的作用，是让系统不只是“能运行”，还要“能被看懂、能被审核、能被管理”。

**背后的功能逻辑**

一套选品系统真正难的地方，不是把规则写进去，而是：

- 谁能看见这些规则
- 规则变了之后谁知道
- 为什么本周这个专题池配额变了
- 为什么某个分数阈值是现在这样

所以策略面板的本质，是把系统当前的判断规则公开出来，让系统成为一个可管理的判断工具，而不是隐藏规则的黑箱服务。

**怎么使用**

进入 `/selection` 页面后，可直接查看当前标准库和生效策略。

**为什么完成这个功能**

因为你要做的是“最强选品系统”，不是“工程师自己知道规则的内部工具”。  
规则层必须产品化，系统才有运营价值。

**当前边界**

当前已经能读策略，但还没有做成前端直接修改策略的后台操作台，这一层后续仍可继续推进。

### 6. 自动标签识别

**完成了什么**

- 新增自动标签识别服务
- 支持自动识别：
  - `season_tags`
  - `holiday_tags`
  - `audience_tags`
  - `scenario_tags`
  - `customization_type`
  - `event_window`
  - `content_hook`
- 商品详情页支持一键自动识别标签
- 批量自动选品流程已接入自动标签识别

**这个功能是干嘛的**

这个功能的作用，是把非结构化商品信息转成结构化标签。

因为后续很多能力都依赖这些标签：

- 专题归类
- 评分
- 人工复核
- 复盘
- 反馈策略

没有标签层，系统就只能靠标题和人工记忆理解商品，扩展性会很差。

**背后的功能逻辑**

商品原始数据本身通常是：

- 标题
- 描述
- 来源平台
- 类目
- 情感总结

这些信息对人来说能看懂，但对规则系统来说不够结构化。  
自动标签识别做的，就是把这些原始文本信号转换成：

- 节日信号
- 人群信号
- 场景信号
- 定制方向信号

也就是说，这个功能本质上是在做：

**原始商品信息 -> 结构化选品特征**

**怎么使用**

- 在商品详情页点击“自动识别标签”
- 或直接运行本周自动选品，由系统批量完成识别

**为什么完成这个功能**

因为标签如果完全依赖人工填写，系统不可能规模化，也不可能支撑后续自动评分和策略机制。

**当前边界**

当前已实现规则增强版自动识别，但还不是 LLM/NLP 深度语义识别版本。  
这一层后续仍可继续加强。

### 7. 人工复核与反馈闭环

**完成了什么**

- 商品详情页支持保存复核反馈
- 支持记录通过、主推确认、淘汰确认、误判原因、后续动作
- 新增反馈汇总能力
- 反馈结果已接入下一轮策略建议和配额调整

**这个功能是干嘛的**

这个功能的作用，是让系统不只会“自动判断”，还会“吸收人工判断结果”。

自动选品系统不可能一开始就完全准确，所以系统必须有一条人工介入和反馈沉淀的通道。

**背后的功能逻辑**

当前逻辑不是把人工复核当成系统外的补丁，而是把它当成系统的一部分：

1. 系统先给出初始判断  
2. 人工确认对不对  
3. 如果不对，记录错因  
4. 系统汇总这些错因  
5. 下一轮专题池策略和配额根据反馈调整

这意味着系统开始具备最基础的“自我修正”能力。

**怎么使用**

- 在商品详情页填写复核反馈并保存
- 在 `/selection` 页面查看反馈汇总和策略建议

**为什么完成这个功能**

因为没有反馈闭环，系统永远只能停留在第一版规则。  
而你要做的是能持续变准的系统，不是一次性的评分器。

**当前边界**

当前已经能记录反馈并给出策略建议，但还没有做到完全自动基于反馈重写召回模型，这部分后续还可以继续推进。

### 8. 选品库批量操作与本周视图

**完成了什么**

- 支持批量更新状态
- 支持批量标记待复核和取消待复核
- 支持按专题、状态、分数、复核状态筛选
- 选品库默认隔离为本周视图
- 修复卡片标签重叠、遮挡图片、排布过挤的问题

**这个功能是干嘛的**

这个功能的作用，是把系统从“自动给结果”推进到“人工能高效处理结果”。

自动选品只是第一步，真正让业务跑起来的是：

- 能快速筛
- 能快速改
- 能快速看本周
- 能快速推进一批商品

**背后的功能逻辑**

本周视图的逻辑，是把历史数据和当前周期分开。  
批量操作的逻辑，是把逐个点击改状态的低效动作，收敛成批量处理流程。

也就是说，这一层不是为了“页面更完整”，而是为了把选品工作真正变成一套可操作的工作流。

**怎么使用**

进入 `/selection/library` 后：

- 用筛选器只看本周商品
- 按状态 / 专题 / 分数 / 复核状态筛选
- 勾选商品后批量更新状态或复核标记

**为什么完成这个功能**

因为自动结果出来以后，如果没有批量处理和本周隔离，人工运营成本仍然很高，系统无法真正被使用。

**当前边界**

当前已经满足第一版实际工作使用，但还没有做更高级的批量理由编辑、批量专题调整等能力。

这个功能的作用，是恢复广告素材浏览能力，让广告数据重新能用于选品判断。

**背后的功能逻辑**

Facebook 广告素材本身不是装饰信息，而是选品判断证据的一部分。  
因为很多商品值不值得推进，不只是看商品信息，还要看：

- 是否有持续投放素材
- 素材呈现方式是什么
- 创意角度是什么
- 是否有礼品化表达

图片不显示，这一整层证据就失效了。

**怎么使用**

进入 `/facebook` 页面即可直接查看素材图片。

**为什么完成这个功能**

因为广告素材库如果不可看，这个模块就失去业务意义。  
修这个问题，本质上是在修复选品证据链，而不是修一个显示问题。

## 四、本周完成后的系统能力变化

本周完成之后，系统能力发生了明显变化。

之前系统更像：

- 商品数据平台
- 收藏页
- 基础看板

现在系统已经开始具备：

- 按周形成候选池的能力
- 对商品做自动判断和分层的能力
- 解释系统判断原因的能力
- 支持人工复核和批量推进的能力
- 把人工反馈反向作用到策略层的能力
- 把权重、阈值、配额、标签来源公开出来的能力

这意味着系统正在从“数据工具”变成“决策工具”。

## 五、本周功能背后的统一逻辑

如果把本周所有功能放在一起看，背后其实只有一条统一逻辑：

**先把商品变成结构化对象，再把结构化对象变成可比较的机会，再把机会变成可执行的本周工作结果，再把人工处理结果回流给策略层。**

可以拆成四步：

1. 商品结构化  
   通过标签识别、专题归类、字段补全，把商品从原始信息变成可分析对象

2. 商品判断化  
   通过评分、理由、状态，把商品从“信息对象”变成“机会对象”

3. 商品工作流化  
   通过本周看板、选品库、批量处理、复核，把商品从“机会对象”变成“可执行任务”

4. 商品反馈化  
   通过反馈记录、反馈汇总、策略建议，把这轮结果变成下一轮优化依据

这就是本周所有功能背后的总逻辑。

## 六、一句话总结

这周不是简单做了几个页面和接口，而是把选品系统从“商品展示系统”推进成了“能按周生成结果、能解释判断、能支持人工复核、能把反馈带回策略层”的第一版选品决策系统。
